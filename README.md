# ESP Triangle Clock

Firmware for the ESP8266-based triangular LED clock. This is a [PlatformIO](https://platformio.org/) project targeting an Adafruit Huzzah ESP8266 board.

## What it does

The clock is a triangular ring of 45 WS2812B LEDs mounted on a assembly that can physically rotate (via a stepper motor + Adafruit Motor Shield). On boot it:

1. Fades the LEDs up and runs self-tests on the temperature sensor, accelerometer, RTC, and motor shield, showing pass/fail status on dedicated LED groups (`updateLEDstatus`).
2. Connects to WiFi and enables OTA updates (`setupOTA`).
3. Uses the onboard accelerometer to physically rotate the clock until it settles pointing straight down under gravity (`pointAccelDown`) — this calibrates its "zero" orientation.
4. Syncs the time from NTP (if WiFi + RTC are available) and rotates to the correct orientation for the current time of day (`initialMove`).

At runtime, the main loop keeps a running clock (incrementing from the RTC-seeded time), periodically rotates the whole clock between three physical orientations depending on time of day (`MORNING` / `MIDDAY` / `NIGHT`), and redraws the LED ring every `LED_FRAMEMS` to reflect the current time — hours/minutes/seconds are each encoded as a set of LEDs that are lit or blacked out (`setLEDsToTime` / `turnOffLEDs`), similar to a binary clock, with the segment meaning depending on which of the three orientations is currently active.

## Layout

```

├── platformio.ini        PlatformIO board/env config + library dependencies
├── src/
│   ├── main.cpp           setup()/loop(), the ticking clock state, and temp-sensor init/read (see below)
│   ├── Motor.cpp           stepper + accelerometer self-leveling/orientation (pairs with include/Motor.h)
│   ├── WiFiOTA.cpp         WiFi/OTA/WebSerial + RTC/NTP time sync (pairs with include/WiFiOTA.h)
│   ├── LEDDisplay.cpp      LED ring rendering + time-to-LED encoding (pairs with include/LEDDisplay.h)
│   └── Debug*.cpp          temporary debug tools, see "Debug tools" section below - to be deleted later
├── include/
│   ├── Motor.h / WiFiOTA.h / LEDDisplay.h   public interface for each src/*.cpp above
│   ├── Debug*.h            public interface for each src/Debug*.cpp above
│   └── wifi_credentials.h (+ .example.h)     see WiFi credentials section below
├── reference/             non-compiled reference material kept for context, see below
├── lib/                   PlatformIO scaffold folder for private/local libraries (unused, default README only)
├── test/                  PlatformIO scaffold folder for unit tests (unused, default README only)
├── .vscode/               editor config auto-generated by the PlatformIO VS Code extension
└── .pio/                  build cache/output (git-ignored, regenerated by `pio run`)
```

The firmware used to live entirely in one 1100-line `main.cpp`. It's now split by subsystem — `main.cpp` keeps just the entry points (`setup()`/`loop()`), the shared "what time is it" ticker state (`hours_now`/`minutes_now`/`seconds_now`), and the temperature sensor (the one subsystem that didn't fit neatly into Motor/WiFiOTA/LEDDisplay). The three module pairs share state across files the way most Arduino sketches do — via `extern` declarations — rather than through a heavier ownership model, since PlatformIO/Arduino projects compile every `.cpp` under `src/` into one firmware image regardless of file boundaries.

## Hardware & libraries

Declared in `platformio.ini`:

| Library | Purpose |
|---|---|
| FastLED | Drives the WS2812B LED ring, color palettes, blending |
| Adafruit Motor Shield V2 | Drives the stepper motor that rotates the clock body |
| Adafruit ADXL343 (+ Adafruit Sensor) | Accelerometer used to find "down" and self-level |
| SparkFun RTC RV8803 | Real-time clock, keeps time across power cycles |
| S5851A | I2C temperature sensor |
| WiiChuck / WiiNunchuck | Nunchuk input (carried over from earlier prototyping, see `Ard-OLEDnunchuk` below) |
| Adafruit GFX + SH110X | OLED display driver (also carried over from prototyping) |
| NTPClient + WiFiUdp | Time sync over WiFi |
| ArduinoOTA | Over-the-air firmware updates |
| WebSerial + ESPAsyncWebServer + ESPAsyncTCP-esphome | Browser-based remote serial console (see below) |
| ArduinoJson, WiFiManager | Pulled in as dependencies, not directly exercised in `main.cpp` today |

Board: `huzzah` (Adafruit Feather HUZZAH ESP8266), framework `arduino`, platform pinned to `espressif8266@^4.2.1` (see build notes below for why).

## Firmware structure

- `setup()` (`main.cpp`) — brings up LEDs, connects WiFi, runs all subsystem init (`setupMotor`, `setupTempSensor`, `setupAccel`, `setupRTC`, `setupOTA`), self-levels the clock, syncs time, and moves to the correct starting orientation.
- `loop()` (`main.cpp`) — services OTA, advances the internal clock once a second, steps the motor if a reorientation is in progress, and redraws the LED ring on a fixed frame interval.

### `main.cpp`

| Function | Description |
|---|---|
| `setupTempSensor()` | Resets and arms the S5851A temperature sensor in low-power shutdown/one-shot mode. Returns `1`/`-1` for success/failure. |
| `readTemp()` | Triggers a one-shot temperature read and logs the result. Currently unused/uncalled — left over from earlier bring-up. |

### `Motor.cpp` / `Motor.h` — stepper, accelerometer, and physical orientation

| Function | Description |
|---|---|
| `setupAccel()` | Initializes the ADXL343 accelerometer at ±2G range. Returns `1`/`-1`. |
| `setupMotor()` | Initializes the Adafruit Motor Shield and sets the starting stepper speed. Returns `1`/`-1`. |
| `pointAccelDown()` | State machine that drives the stepper back and forth, reading the accelerometer, until the assembly comes to rest pointing straight down (gravity-referenced zero point). A more refined alternative algorithm from the merged `espTriangleClock-Home` project is kept as plain reference text in `reference/pointAccelDown_home_algorithm.txt` (not compiled), pending testing on real hardware. |
| `readAccel(numOfReads)` | Averages `numOfReads` accelerometer Y-axis samples and returns the mean. |
| `initialMove()` | Performs the first physical move to the correct orientation for the current hour, right after `pointAccelDown()` finishes. |

Key definitions: `NUMOFMOTORSTEPS` (1024, steps for a full reorientation move), `MORNING`/`MIDDAY`/`NIGHT` (the three physical orientations), `RPM_START`/`HISPEED`/`MEDSPEED`/`SLOWSPEED`/`STARTSPEED` (stepper speeds used during self-leveling, chosen by how close the accelerometer reading is to "down"). Exposes `AFMS`/`myMotor` (motor shield/stepper objects) and `orientationNow`/`orientationStepCounter`/`orientationDirectionMove` externally — `main.cpp`'s `loop()` and `LEDDisplay.cpp`'s `setLEDsToTime()` both read/drive these to carry out reorientation moves a step at a time.

### `WiFiOTA.cpp` / `WiFiOTA.h` — WiFi, OTA, WebSerial, RTC, NTP

| Function | Description |
|---|---|
| `beginWiFiConnection()` | Sets WiFi to station mode and kicks off the primary-network connection attempt (non-blocking). |
| `setupRTC()` | Initializes the RV8803 RTC, reads the current time into `hours_now`/`minutes_now`/`seconds_now`, and configures its periodic update interrupt. Returns `1`/`-1`. |
| `setupOTA()` | Waits for the WiFi connection, falling back to a second set of credentials, then registers `ArduinoOTA` callbacks and starts the WebSerial console (see below). Returns `1`/`-1`. |
| `updateTime()` | Fetches the current time via NTP and writes it into the RTC. |
| `logPrint(value)` / `logPrintln(value)` / `logPrintln()` | Print helpers used everywhere `Serial.print`/`println` used to be called directly — write to `Serial` and, once WiFi/WebSerial are up, mirror to the WebSerial browser console (see below). |
| `logPrintf(fmt, ...)` | `printf`-style variant of the above, used by the two `ArduinoOTA` progress/error callbacks. |
| `onWebSerialMessage(data, len)` | Callback for text typed into the WebSerial browser console — currently just echoes it back via `logPrintln`. |

Key definitions: `DAYLIGHTSAVINGS` (manual DST offset flag used by `updateTime()`). Exposes `rtc` (RV8803 object) and `webSerialReady` externally — `LEDDisplay.cpp`'s `updateLEDs()` reads both.

### `LEDDisplay.cpp` / `LEDDisplay.h` — LED ring rendering

| Function | Description |
|---|---|
| `updateLEDstatus()` | Animates the status-indicator LED groups (temp/RTC/motor/accel/wifi) to reflect pass/fail of each subsystem check. |
| `updateLEDs()` | Repaints the full LED ring to a base color, then calls `setLEDsToTime()` to blank out the segments that encode the current time. |
| `setLEDsToTime()` | Core time-display logic — decides whether a reorientation move is needed (morning/midday/night) and blacks out the specific LEDs that encode the current hour/minute/second in the active orientation. |
| `turnOffLEDs(startIndex, endIndex)` | Sets an inclusive LED index range to black. |

Key definitions: `NUM_LEDS` (45), `LED_PIN` (13), `LED_TYPE`/`COLOR_ORDER`, `LED_FRAMEMS` (redraw interval in ms), `BRIGHTNESS`, `passFail_gp`/`currentPalette` (gradient palette used to color the subsystem status LEDs during startup self-test). Exposes `leds[]` externally — `Motor.cpp`'s `pointAccelDown()` writes directly into it for the self-leveling flash animation.

Pin definitions `BUTTON_A`/`BUTTON_B`/`BUTTON_C`, `SW_TOG`, `RTCINT_PIN` live in `main.cpp` (carried over from the nunchuk/OLED prototype; only `SW_TOG` is actually wired up today).

## WiFi credentials

`WiFiOTA.cpp` pulls its primary/backup WiFi SSID and password from `include/wifi_credentials.h`, which is git-ignored so real credentials never get committed. To set up a fresh checkout:

1. Copy `include/wifi_credentials.example.h` to `include/wifi_credentials.h`.
2. Fill in `WIFI_PRIMARY_SSID` / `WIFI_PRIMARY_PASSWORD` and `WIFI_BACKUP_SSID` / `WIFI_BACKUP_PASSWORD`.

`beginWiFiConnection()` connects to the primary network first; `setupOTA()` falls back to the backup network if the primary connection fails.

Both networks referenced here (`JAMES15` and `BecauseFi`) were shared across the two physical clock units (`espTriangleClock` and the now-merged `espTriangleClock-Home`) — only the primary/backup priority order differed between them. If you're flashing the unit that used to live in `espTriangleClock-Home`, you may want `BecauseFi` as primary instead.

## Remote debugging (WebSerial)

Once WiFi connects, the firmware serves a browser-based serial console at `http://<device-ip>/webserial` (via [WebSerial](https://github.com/ayushsharma82/WebSerial) + ESPAsyncWebServer), so you can watch debug output — including the whole `pointAccelDown()` self-leveling state machine — without a USB cable, e.g. over the same OTA connection you already use to flash it. The IP address is printed once to `Serial` right after connecting (`setupOTA()`).

Nearly every existing `Serial.print`/`println`/`printf` call was swapped for `logPrint`/`logPrintln`/`logPrintf` (see helper table above), which write to both `Serial` and WebSerial. `updateLEDs()` is the one exception worth knowing about: it logs the time every `LED_FRAMEMS` (10ms), so its WebSerial mirror is throttled to ~1/sec to avoid flooding the websocket — it still logs to `Serial` at full rate as before. Typing in the WebSerial browser console (or physical Serial) both echo back what you sent and get parsed as a command — see "Debug tools" below.

**Build notes, if you're touching includes or dependencies here:**
- The platform version had to be pinned to `espressif8266@^4.2.1` (it was previously unpinned, resolving to 3.2.0) — older ESP8266 Arduino cores have a bug ([esp8266/Arduino#7249](https://github.com/esp8266/Arduino/issues/7249)) where `ESP8266WiFi.h`'s `wl_tcp_state` enum collides with lwIP's own TCP state enum once something (here, ESPAsyncWebServer) pulls in `lwip/tcp.h`.
- `ESPAsyncWebServer.h`/`WebSerial.h` must be included before `ESP8266WiFi.h` — WebSerial's own header includes `ESP8266WiFi.h` internally, so this only works because `WiFiOTA.h` lists them in that order itself, and every other file that needs any of the three (`main.cpp`, `Motor.cpp`, `LEDDisplay.cpp`) reaches them by including `WiFiOTA.h` rather than including `ESP8266WiFi.h` directly.
- The always-unused `ESP8266WebServer.h`/`DNSServer.h`/`ESP8266mDNS.h` includes (leftover from the pre-split `main.cpp`) were dropped rather than carried into `WiFiOTA.h` — `ESP8266WebServer.h` defines its own `HTTP_GET`/`HTTP_POST`/etc. enum that's ambiguous with ESPAsyncWebServer's identically-named one once both are visible in the same translation unit.
- WebSerial is licensed AGPL-3.0 (stronger copyleft than this project's other MIT/LGPL dependencies) — only relevant if this firmware is ever distributed rather than just flashed to your own hardware.
- Flash/RAM usage on the `huzzah` build grew from 33.3%/38.8% to 39.8%/44.8% (of 1,044,464 B flash / 81,920 B RAM) — still well under half, plenty of headroom left.

## Debug tools (temporary — will be removed once tuning is done)

Typed into either the physical Serial monitor or the WebSerial browser console (one command per line), these three mutually-exclusive modes each take over a different slice of the firmware for hands-on tuning. `DebugConsole.cpp` reads the input and tracks which mode (if any) is active; `main.cpp`'s `loop()` checks that mode and either fully hands off to the active tool or (for `TIMETEST`) just swaps out the normal ticker. From `DEBUG_NONE`, send `EXIT` any time to leave whichever mode is active and resume normal operation.

| Command | Mode | What it does |
|---|---|---|
| `LEDTEST` | `Motor`/time untouched | Ignores the real clock entirely and cycles the whole 45-LED ring through 10 hue steps at full saturation/brightness, so you can confirm every LED lights up and check colors independent of the time-encoding logic. Sub-commands (send while active): `MINUTES` toggles every other LED in the minutes group (21-32) on/off; `SECONDS` does the same for the seconds group (33-44); `HOURS` cycles the three 7-LED hour groups (0-6/7-13/14-20) through all-on → `0,2,4,6` → `1,4` → all-off → repeat. |
| `TIMETEST` | LEDs/motor keep running normally | Pauses NTP/RTC as the time source. Send `hh:mm:ss` (e.g. `14:30:00`) to jump straight to a time, or `FAST-N` (e.g. `FAST-100`) to advance the clock N seconds per real second instead of 1 — handy for watching the day cycle (and the 3x/day motor reorientation) play out quickly. `setLEDsToTime()`/`updateLEDs()` and the orientation-driven motor stepping in `loop()` are untouched, so they keep responding to whatever `hours_now`/`minutes_now`/`seconds_now` this mode sets. Exiting resyncs from the RTC via `setupRTC()`. |
| `ACCELTEST` | LEDs/time untouched | No LED or time changes — just the accelerometer and motor, to help tune `pointAccelDown()`. While idle, reports an instantaneous accelerometer reading plus a ~5-second rolling average roughly twice a second. Send `+N` / `-N` (e.g. `+50`) to step the motor N steps forward/backward one at a time (deliberately slow — a 20ms delay between steps), printing a `[MOVING]`-tagged reading after every single step so you can correlate motor motion with the accelerometer response in real time. |

Each mode's state and command parsing lives entirely in its own `Debug*.cpp`/`.h` pair (`DebugLedTest`, `DebugTimeTest`, `DebugAccelTest`), plus `DebugConsole.cpp` for routing. The only touches to the permanent files are a handful of lines marked `// DEBUG TOOL` in `main.cpp` (the includes, the startup hint, and the `loop()` mode dispatch) and `WiFiOTA.cpp` (one line in `onWebSerialMessage` routing incoming text to the command parser). To remove these tools later: delete `Debug*.cpp`/`Debug*.h` and every line tagged `// DEBUG TOOL`.

A couple of implementation notes: `ACCELTEST`'s `+N`/`-N` move is non-blocking — one microstep per `debugAccelTestTick()` call, paced by `millis()` rather than `delay()`, the same way `main.cpp`'s `loop()` steps the orientation motor one microstep per iteration instead of blocking through a whole reorientation. That keeps `ArduinoOTA`/WebSerial/Serial serviced between steps, so `[MOVING]` readings stream in live instead of arriving all at once after the move finishes. The `MINUTES`/`SECONDS` "half on/half off" pattern blacks out every *other* LED in the group (alternating), not a contiguous first-half/second-half split — easy to change in `DebugLedTest.cpp` if you wanted the other pattern instead.

## `reference/` — non-compiled reference material

`reference/pointAccelDown_home_algorithm.txt` holds the alternate self-leveling algorithm from the merged `espTriangleClock-Home` project (see merge notes below). It's plain text, not `.cpp`/`.h`, so PlatformIO never compiles it — it's kept only so the algorithm isn't lost, in case it's worth adapting into `Motor.cpp`'s `pointAccelDown()` after testing on real hardware.

## Notes for future cleanup

- The alternate self-leveling algorithm in `reference/pointAccelDown_home_algorithm.txt` needs testing against real hardware before deciding whether to adopt it in `Motor.cpp`'s `pointAccelDown()`.
- `NUMOFMOTORSTEPS` is 1024 here; the merged `espTriangleClock-Home` project had used 200 for the same stepper init call. That's a steps-per-revolution value tied to the physical motor/gearbox on each unit — worth double-checking against whichever hardware you're flashing before relying on it.

## `espTriangleClock-Home` merge

There used to be a second, near-duplicate PlatformIO project, `espTriangleClock-Home`, built for a second physical clock unit at a different location. It's been merged into this project and removed. What it had that this project didn't, and how each was resolved:

- **WiFi credentials** — same two networks as this project, just prioritized in the opposite order (see the WiFi credentials section above).
- **Stepper steps-per-revolution** — used `200` instead of `NUMOFMOTORSTEPS` (1024); kept this project's `1024` as the default (see cleanup notes above).
- **Reworked self-leveling algorithm** — Home's `pointAccelDown()` had a materially different/more refined implementation (rolling-average + derivative final-tuning stage, longer timeout, a no-sensor fallback). Carried over as plain reference text in `reference/pointAccelDown_home_algorithm.txt` for future testing, rather than replacing the active one in `Motor.cpp` outright.
- **OLED + Nunchuk debug UI** — Home routed setup/status logging to an SH1107 OLED display (plus Wii Nunchuk input) instead of `Serial`. Dropped entirely in the merge — not carried over.
- **No time-of-day rotation** — Home's `setLEDsToTime()` had no concept of the `MORNING`/`MIDDAY`/`NIGHT` physical rotation this project uses; it only ever displayed time in a single fixed orientation. This project's rotation behavior was kept as-is.
- `platformio.ini` upload settings (`upload_protocol = espota` to a different IP) were device-specific to that physical unit and weren't carried over — this project still uploads over serial by default.
